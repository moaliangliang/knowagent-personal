## 1. 产品战略

### 1.1 一句话定位

> **知行的本质不是「101 个命令」，而是一个对话界面 + 一组高频场景。**

用户打开知行，看到的是一个干净的对话窗。输入自然语言，AI 理解意图后执行操作。不需要学命令，不需要配环境。

### 1.2 目标用户

**普通办公室白领** —— 非技术背景，每天有重复性电脑操作，想用 AI 提升效率但不具备技术能力。

### 1.3 核心场景（Phase 1）

| 场景 | 用户需求 | AI 做什么 | 系统操作 |
|------|---------|-----------|---------|
| 📝 日报助手 | "帮我写今天的日报" | 读日历+邮件+待办 → 生成日报草稿 | Calendar API / Mail / TODO |
| 📁 文件整理 | "把桌面图片按日期归档" | 理解分类规则 | 文件系统操作 |
| 📧 快捷邮件 | "回复张三的邮件说收到" | 提取邮件内容 → 生成回复 | Mail.app API |
| 📅 会议摘要 | "今天下午的会准备什么" | 读日历事件+附件 → 输出要点 | Calendar API |
| 🌐 AI 助手 | "翻译这段" / "润色一下" | 直接 AI 处理 | 剪贴板读写 |

### 1.4 商业模式

| 层级 | 价格 | 权益 |
|:----|:----:|:----|

## 2. 产品交互设计

### 2.1 核心交互模式

交互方式只有一种：**对话**。

首屏展示 4 个推荐场景按钮，用户可直接点击或打字输入。

```
你好！我是知行，你的桌面 AI助手

推荐场景：
[📝 写日报] [📁 整理桌面] [📧 快捷邮件] [🌐 翻译]

─────────────────────────────────
│ 帮我写今天的日报            │ ➤ │
─────────────────────────────────
```

### 2.2 关键设计原则

1. **零配置**：下载 → 打开 → 开始对话。没有终端、没有 Python、没有 config.yaml
2. **场景推荐**：首屏直接给推荐场景按钮，用户不知道能做什么时直接点击
3. **渐进式披露**：先解决一个具体问题，深度使用后再展示更多能力
4. **所见即所得**：AI 执行前展示计划，确认后再执行

### 2.3 Chat UI 组件

| 组件 | 说明 |
|:----|:------|
| 对话气泡列表 | 用户消息 + AI 回复，markdown 渲染 |
| 场景推荐按钮 | 首屏 4 个大按钮，点击自动填充输入 |
| 输入框 | 自动拉伸 textarea，Enter 发送，Shift+Enter 换行 |
| 打字动画 | AI 思考时显示 "⚡ 知行思考中..." |
| 交互卡片 | AI 生成的可操作卡片（如日报预览+「复制/发送邮件」按钮） |
| 对话历史 | 本地持久化，启动恢复 |

## 3. 技术架构

### 3.1 架构总览

```
┌─────────────────────────────────┐
│   Chat UI (Electron 新窗口)      │
│   对话气泡 + 场景推荐 + 输入框    │
└────────────┬────────────────────┘
             │ 对话文本 (WebSocket)
┌────────────▼────────────────────┐
│   AI Orchestrator (新增)         │
│   意图解析 → 步骤拆解 → 工具调用  │
│   → 结果合成 → 回复              │
└────────────┬────────────────────┘
             │ 调用 cmd_* 函数
┌────────────▼────────────────────┐
│   server.py + zhishing (已有)    │
│   101 个 cmd_* 函数，零改动       │
└─────────────────────────────────┘
```

### 3.2 核心新增：AI Orchestrator

**文件**: `zhixing/agent/orchestrator.py`

核心流程：

```
用户输入 → ① LLM 解析意图
          → ② LLM 决定调用哪些工具
          → ③ 执行工具（调用现有的 cmd_* 函数）
          → ④ 工具结果送回 LLM
          → ⑤ LLM 合成最终回复
          → ⑥ 返回到 Chat UI
```

工具定义直接从 `tools.py` 的 `get_tool_definitions()` 复用。

### 3.3 路径分类

| 类型 | 示例 | 处理方式 |
|:----|:----|:--------|
| 单意图单步 | "系统状态" | 调 1 个函数，直接返回 |
| 单意图多步 | "写日报" | 调多个函数，LLM 合成 |
| 意图不明确 | "帮我弄一下" | LLM 反问澄清 |
| 超出能力 | "帮我修图" | LLM 说明不支持 |
| 纯对话 | "今天天气怎么样" | LLM 直接回复（不调工具） |

### 3.4 用量限制器

```
用户 → usage_tracker.add() → 查本月使用次数
  → 免费用户 >= 50 → 拦截 + 升级提示
  → Plus 用户 → 放行
```

存储: `~/.zhixing/usage.db` (SQLite)
云端可选: 配合知行 API 服务可实现多设备同步

### 3.5 AI API 接入

| 模式 | 配置方式 | 适用阶段 |
|:----|:--------|:--------|
| 用户自备 Key | 设置界面输入 DeepSeek/OpenAI Key | 高级用户 |
| 知行托管 | 默认开通，送 50 次/月 | 默认，Phase 1 MVP |

### 3.6 零改动部分

- `server.py` —— 不变
- `zhixing/agent/tools.py` —— 101 个命令全保留
- Chrome 扩展 —— 不变
- 构建/发布流程 —— 不变

## 5. 风险与缓解

| 风险 | 概率 | 影响 | 缓解方案 |
|:----|:----:|:----:|:--------|
| LLM 调用失败/超时 | 中 | 高 | 重试机制 + 降级到固定回复 |
| 工具调用幻觉（LLM 编造参数） | 中 | 中 | 工具参数类型校验 + 执行前确认 |
| 用户隐私顾虑 | 低 | 高 | 本地优先 + 明确隐私政策 |
| API 成本超出预期 | 低 | 中 | 硬性次数上限 + 用量监控告警 |
| 用户不习惯对话交互 | 中 | 中 | 场景推荐按钮降低上手门槛 |
