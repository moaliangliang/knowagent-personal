## 1. 工作流是什么

工作流将多个系统命令按顺序编排，一键执行。例如：

```
"每天9点检查系统 → 检查电池 → 查看日程 → 发通知"
```

**之前**：要手动敲 4 条命令
**之后**：`zhi workflow` 选模板 / 一句话描述 / 问答引导 — 自动执行

### 适用场景

| 场景 | 示例 |
|:----|:-----|
| 每日检查 | 开机后一键检查系统、电池、网络、日程 |
| 定时监控 | 每 30 分钟检查一次 CPU 温度、磁盘空间 |
| 批量操作 | 依次播放周杰伦→王力宏→林俊杰 |
| 自动化流程 | 连接 VPN → 确认 IP → 测速 → 通知 |
| 下班收尾 | 检查待办 → 保存剪贴板 → 静音 → 停止音乐 |

## 2. 三种配置方式

### 方式一：一句话描述（最快）

```bash
zhi workflow_create text="每天9点检查系统然后发通知"
```

系统自动解析出步骤：

```
🧠 从描述: 「每天9点检查系统然后发通知」
  解析出 3 步:
  1. ⏰ 每天9点触发
  2. 系统状态
  3. 通知

📋 直接运行: run=true
📋 保存为模板: save=true name=名称
```

**支持的自然语言模式**：

| 你说 | 它理解为 |
|:----|:---------|
| `每天9点` / `每隔30分钟` | 触发条件（定时/间隔） |
| `检查系统` / `检查电池` / `截图` | 对应命令 |
| `播放XX的歌` | `music_search_online` |
| `通知XX` / `发通知` | `notification` |
| `然后` / `接着` / `再` | 步骤分割 |
| `等待5秒` / `等待2分钟` | `wait` 步骤 |
| `音量50` | `system_volume` |

### 方式二：交互引导（推荐新手）

```bash
zhi create_wf
```

进入问答模式，一步步配置：

```

📌 工作流名称: 每日检查

⏰ 触发方式:
  1. 手动执行（默认）
  2. 定时执行（每天固定时间）
  3. 间隔执行（每隔 N 秒）
请选择 (1-3): 2
  每天几点? (0-23): 9

📋 添加步骤:  (输入数字快捷选择，空行结束)
快捷选择: 1=系统状态 2=电池 3=日程 4=待办 5=截图 6=通知 7=播放 8=网络 9=磁盘 0=番茄钟
步骤 1: 1
  ✅ 已添加
步骤 2: 6
  通知内容: 早上好！
  ✅ 已添加
步骤 3: (回车结束)

📋 预览 → [1]保存 [2]运行 [3]保存并运行
```

## 4. 高级参数详解

每个步骤除了基本的 `cmd` 和 `params`，还支持以下高级参数：

```yaml
steps:
  - cmd: system_status
    desc: "系统状态"
    wait: 2           # ⏱ 执行后等待秒数（默认0.5，设为0不等待）
    loop: 5           # 🔁 重复执行次数
    retry: 2          # 🔂 失败后重试次数
    timeout: 30       # ⏰ 单步超时秒数（超时算失败）
    on_error: stop    # 🚨 失败策略: skip(跳过继续) / stop(终止)
    if: "${result}包含OK"  # 🔀 条件执行（满足才运行）
    sub_workflow: "系统报告" # 🧩 调用预设工作流

  - cmd: notification
    params:
      text: "${result_system_status}"  # 📦 引用上一步输出
```

### 4.1 🔁 循环 (loop)

```bash
# 通知弹 5 次
zhi wf_exec steps='[{"cmd":"notification","params":{"text":"hi"},"loop":5}]'

# 每 10 分钟检查一次系统，共 6 次
zhi wf_exec steps='[{"cmd":"system_status","loop":6,"wait":600}]'
```

### 4.2 🔂 重试 (retry) + ⏰ 超时 (timeout)

```bash
# 网络请求失败重试 2 次，每次最多等 10 秒
zhi wf_exec steps='[{"cmd":"my_ip","retry":2,"timeout":10}]'

# 关键步骤：失败就终止
zhi wf_exec steps='[{"cmd":"vpn_status","retry":3,"timeout":5,"on_error":"stop"}]'
```

### 4.3 🧩 子工作流 (sub_workflow)

```bash
# 在一个工作流中调用另一个预设模板
zhi wf_exec steps='[
  {"sub_workflow":"🌐 网络诊断"},
  {"cmd":"notification","params":{"text":"网络检查完成"}}
]'
```

### 4.4 📦 变量传递 (${var})

步骤之间的结果可以通过 `${变量名}` 引用：

```yaml
steps:
  - cmd: system_status       # 输出自动保存到 ${result_system_status}
  - cmd: notification
    params:
      text: "${result_system_status}"  # 引用上一步结果
```

### 4.5 🏃 后台执行 (bg)

```bash
# 后台执行，不阻塞终端
zhi wf_exec bg=true steps='[{"cmd":"system_status","loop":24,"wait":3600}]'
```

## 5. 自定义模板

### 保存自定义模板

```bash
# 1. 先用自然语言创建工作流
zhi workflow_create text="检查系统然后查天气再发通知" run=true

# 2. 保存为模板
save_template 我的晨检
```

或通过引导模式创建后，在最后一步选择 **保存**。

### 查看自定义模板

```bash
zhi workflow
```

自定义模板会出现在预设列表末尾。

### 自定义模板持久化

自定义模板保存在 `~/.zhixing/custom_templates.json`，可手动编辑、备份、分享：

```json
{
  "📌 我的晨检": [
    {"cmd": "system_status", "desc": "系统状态"},
    {"cmd": "calendar", "desc": "今日日程"},
    {"cmd": "notification", "params": {"text": "检查完成"}, "desc": "通知"}
  ]
}
```

### 分享模板

```json
// 分享 ~/.zhixing/custom_templates.json 给对方
// 或分享 YAML 格式
```

## 7. 常见问题

### Q: 模板步骤限制？

**Community 版**：最多 5 步
**Pro 版**：无限步骤

### Q: 如何分享工作流？

1. 导出 YAML：在工作流编辑器中点击 **📋 导出 YAML**
2. 分享 `~/.zhixing/custom_templates.json` 给对方

### Q: 自定义模板会丢失吗？

不会。保存在 `~/.zhixing/custom_templates.json`，卸载重装后还在。重装知行后导入即可。

### Q: 子工作流可以嵌套吗？

可以。子工作流中的步骤也可以调用其他子工作流（递归），注意避免循环引用。

### Q: 后台执行的工作流怎么查看结果？

后台执行使用 `bg=true`，结果不显示在终端。建议在工作流最后加 `notification` 步骤来通知完成状态。

### Q: 简写命令有哪些？
